# 보안 운영 규칙 (SECURITY.md)

> **이 저장소(`wxp-mkt`)는 PUBLIC 입니다.**
> 여기 올라간 모든 것은 전 세계에 공개되며, **한 번 올라가면 파일을 지워도 되돌릴 수 없습니다**
> (git 히스토리·fork·검색 캐시에 이미 복제됨). 그래서 **삭제(대응)보다 예방이 훨씬 중요**합니다.

---

## 0. 대원칙

**"전 세계에 공개돼도 100% 괜찮은 것만 올린다."**

이 저장소의 용도는 **대외 공개용 정적 콘텐츠(마케팅/데모 페이지) 전용**입니다.
고객정보·내부코드·실데이터가 조금이라도 닿는 작업은 **별도의 Private 저장소**에서 합니다.

---

## 1. 절대 올리면 안 되는 것 (금지 목록)

- **고객 개인정보** — 이름·연락처·이메일·주민번호·이력서, 잡코리아/알바몬/나인하이어/잡플래닛/게임잡 회원·이용자 데이터 일체
- **인증정보** — API 키, 액세스 토큰, 비밀번호, DB 접속정보, `.env`
- **실데이터** — DB 덤프, CSV/엑셀 원본, 서비스 로그, 쿼리 결과
- **내부 인프라 정보** — 내부 URL·서버 주소·아키텍처 문서·사내 시스템 설명

---

## 2. 예방 프로세스 (다중 안전망)

| 단계 | 장치 | 역할 |
|---|---|---|
| ① 진입 차단 | `.gitignore` | 위험 파일이 커밋에 잡히지 않게 사전 차단 |
| ② 자동 탐지 | GitHub Secret Scanning + Push Protection | push 순간 키·토큰 자동 감지·차단 |
| ③ 인간 검토 | Branch Protection + PR | `main` 직접 push 금지 → 브랜치 → PR → 확인 후 병합 |

### 커밋 전 셀프 체크리스트
- [ ] 이 내용이 전 세계에 공개돼도 괜찮은가?
- [ ] 고객정보·개인정보가 단 한 줄도 없는가?
- [ ] 키·토큰·비밀번호·접속정보가 없는가? (주석·예시 포함)
- [ ] 실데이터 파일(csv/xlsx/sql/db 등)이 섞여 있지 않은가?
- [ ] 내부 URL·서버 주소가 노출되지 않는가?

---

## 3. 작업 흐름 (Branch → PR → Merge)

```
main 에 직접 push 금지.
1) 브랜치 생성:   git switch -c feature/무엇
2) 작업 + 커밋
3) push + PR 생성: gh pr create
4) 리뷰/체크 후 병합
```

---

## 4. 사고 대응 절차 (실수로 올렸다면)

1. **"이미 유출됐다"고 가정** — 노출된 키·비밀번호·토큰은 **즉시 폐기·재발급** (히스토리 삭제보다 이게 최우선)
2. 해당 파일/커밋을 히스토리에서 제거 후 강제 push
3. 무엇이·언제·어디까지 노출됐는지 기록하고 필요 시 보안 담당자에게 보고
